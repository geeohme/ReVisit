# Spaces Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a top-level "Space" container above categories so every bookmark belongs to exactly one Space and categories become scoped per Space. Per-installation Space selection (enabled set, default, last-used) lives in a NEW non-synced `chrome.storage.local` key `rvLocal`; Space definitions, the new `spaceId` on categories/bookmarks, and the new `rvData.spaces` array sync through the existing flat per-record LWW engine. Source of truth is the approved spec at `docs/superpowers/specs/2026-06-04-spaces-design.md`.

**Architecture:** No-build Manifest V3 Chrome extension. Flat repo layout — every source `*.js` and test `*.test.js` lives at the repo ROOT; `db/schema.sql` is the only file in a subdir we touch. The sync engine (`rv-sync-core.js` pure logic + `sync.js` Supabase client) only ever reads/writes the `rvData` and `rvTranscripts` storage keys, so anything added to `rvData` syncs and anything in the new `rvLocal` key is installation-private by construction. Pure, testable logic goes in `rv-sync-core.js` and a NEW `rv-spaces-core.js` (both using the dual `self.X = ...` + `module.exports` export pattern); DOM wiring in `list-modal.js`/`content.js`/`onboarding.js` is implemented directly and verified manually (it cannot be unit-tested with `node:test`).

**Tech Stack:** Vanilla JS (service worker + content scripts + extension pages), Supabase (GoTrue auth + PostgREST), Postgres with RLS. Tests: Node built-in runner — `const test = require('node:test'); const assert = require('node:assert');`. Run all: `npm test` (= `node --test`). Single file: `node --test rv-spaces-core.test.js`. Single test: `node --test --test-name-pattern="<name>"`.

## Reserved constants & shared names (use these EXACTLY everywhere)

- Reserved migration Space id: the string `"default-space"`.
- `rvData.spaces` element shape: `{ id, name, priority, updatedAt, deletedAt }`. Sync identity key = `'id'`.
- `rvData.categories` element shape gains `spaceId`: `{ spaceId, name, priority, updatedAt, deletedAt }`. Identity key = composite `c.spaceId + ' ' + c.name`.
- `rvData.bookmarks` element gains `spaceId` (string). Sync identity stays `'id'`.
- NEW local key `rvLocal` shape: `{ enabledSpaceIds: string[], defaultSpaceId: string, lastUsedListSpaceId: string }`. These three field names are spelled exactly `enabledSpaceIds`, `defaultSpaceId`, `lastUsedListSpaceId` in every file.
- The category composite-key helper is named `catKey(c)` and returns `c.spaceId + ' ' + c.name`. Defined once per consuming file (or read off `RvSpacesCore`).
- Backup file version: `3`.

  function _keyOf(key, rec) { return typeof key === 'function' ? key(rec) : rec[key]; }

    const map = new Map(localList.map(r => [_keyOf(key, r), r]));

  // Decide which setup flavor to run on list-page load.
  //   'none'    — rvLocal.defaultSpaceId is set, in enabledSpaceIds, and points at a LIVE Space.
  //   'migrate' — setup needed AND rvData.spaces has no live Space (pre-Spaces upgrade → Flavor A).
  //   'pick'    — setup needed AND live Spaces already exist (second-browser → Flavor B).
  function setupGateDecision(rvData, rvLocal) {

    if (ok) return 'none';
    return live.length === 0 ? 'migrate' : 'pick';
